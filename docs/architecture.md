> rules for the tricky domain logic (scheduling & streaks). No hand-waving: if a rule has
> an edge case, it is spelled out here.

## 1. Stack (decided)

> Versions track the **installed project** — Expo **SDK 57**, React **19.2**, React Native
> **0.86**, Reanimated **4.5**. `package.json` is the source of truth; see
> [library-docs.md](./library-docs.md) for the full pinned table. The repo is **already
> scaffolded** (Expo Router app), so Phase 0 is reconciliation, not `create-expo-app`.

| Concern | Choice | Why |
| --- | --- | --- |
| Framework | **Expo (SDK 57)**, React Native 0.86, React 19.2, TypeScript | Managed workflow, OTA, native modules without ejecting. |
| Navigation | **Expo Router** (file-based, typed routes) | File = route. Native bottom tabs. Type-safe links. |
| Styling | **NativeWind v4** + custom Material 3 tokens | Tailwind DX; tokens wired as CSS variables so light/dark is one switch. |
| Local DB | **expo-sqlite** | On-device SQL, fast, DevTools inspector. |
| ORM / queries | **Drizzle ORM** (`drizzle-orm/expo-sqlite`) | Type-safe schema + `useLiveQuery` reactive reads. |
| Migrations | **drizzle-kit** → `useMigrations` at boot | Versioned schema, survives app updates. |
| Animation | **react-native-reanimated** 4.5 (+ `react-native-worklets`) + **gesture-handler** | Required for the Expressive spring/morph motion. |
| Dates | **date-fns** (+ `date-fns-tz` if needed) | Deterministic, tree-shakeable date math. |
| State (UI) | React state + Drizzle live queries; **zustand** for cross-screen prefs | Server-state = the DB. UI/pref state = zustand. No Redux. |

There is **no** backend, API layer, auth, or network client. Do not add one.

**UI direction (decided):** we hand-build the Material 3 Expressive system with **NativeWind
+ custom tokens** ([ui-rules.md](./ui-rules.md), [ui-tokens.md](./ui-tokens.md)). The
scaffold currently includes `@expo/ui`, `expo-glass-effect`, and `expo-symbols` — these are
**not part of the v1 plan** and should be removed (or left unused) rather than mixed into
the custom system. `expo-symbols`/SF Symbols may stay only if used purely for native-tab
icons; do not build content UI on the native/glass components.

## 2. Guiding architectural principles

1. **The database is the state.** Screens subscribe to the DB via `useLiveQuery`. There is
   no separate in-memory cache of habits/check-ins to keep in sync.
2. **Derive, don't store, what you can compute cheaply.** Streaks and completion rates are
   *computed from check-ins*, never stored as columns that can drift. (Exception: an
   optional cached streak for perf is allowed only behind a pure recompute function.)
3. **Pure domain logic, isolated from React.** All scheduling/streak math lives in
   `src/domain/` as pure functions taking plain data + a reference date. They are unit-
   testable with no DB and no React.
4. **Time is an input, never a global.** Every function that cares about "today" takes a
   `now: Date` (or `today: string`) parameter. Never call `new Date()` deep inside domain
   logic — it makes tests non-deterministic and hides timezone bugs.

**Hard rule:** files under `src/app/` are *thin*. They compose components from `src/ui` and

## 4. Data model (Drizzle schema)

Defined once in `src/db/schema.ts`. Dates are stored as **ISO-8601 strings** in the local
calendar sense: check-in days are `YYYY-MM-DD` (no time, no zone), timestamps are full ISO.

```ts
import { sqliteTable, text, integer, uniqueIndex, index } from 'drizzle-orm/sqlite-core';
import { sql } from 'drizzle-orm';

export const habits = sqliteTable('habits', {
  id: text('id').primaryKey(),                     // uuid (see lib/id.ts)
  name: text('name').notNull(),
  color: text('color').notNull(),                  // token key, e.g. 'green' — NOT a hex
  icon: text('icon').notNull(),                    // icon name from the icon set
  // Cadence, stored flat to stay queryable:
  cadenceType: text('cadence_type', {
    enum: ['daily', 'weekdays', 'weekly_count'],
  }).notNull(),
  weekdays: text('weekdays'),                      // CSV '1,3,5' (0=Sun..6=Sat) when weekdays
  weeklyTarget: integer('weekly_target'),          // N when weekly_count
  sortOrder: integer('sort_order').notNull().default(0),
  archivedAt: text('archived_at'),                 // ISO ts or null
  createdAt: text('created_at').notNull().default(sql`(CURRENT_TIMESTAMP)`),
});

export const checkins = sqliteTable(
  'checkins',
  {
    id: text('id').primaryKey(),
    habitId: text('habit_id')
      .notNull()
      .references(() => habits.id, { onDelete: 'cascade' }),
    day: text('day').notNull(),                    // 'YYYY-MM-DD' local calendar day
    createdAt: text('created_at').notNull().default(sql`(CURRENT_TIMESTAMP)`),
  },
  (t) => ({
    // One check-in per habit per day — enforced at the DB level:
    uniqDay: uniqueIndex('uniq_habit_day').on(t.habitId, t.day),
    byHabit: index('idx_checkins_habit').on(t.habitId),
    byDay: index('idx_checkins_day').on(t.day),
  }),
);

export type Habit = typeof habits.$inferSelect;
export type NewHabit = typeof habits.$inferInsert;
export type Checkin = typeof checkins.$inferSelect;
```

**Design decisions embedded above (do not "fix" without a reason):**

- `color`/`icon` store **token keys**, not raw values, so re-theming never requires a
  data migration.
- Cadence is stored **flat** (not JSON) so we can filter in SQL later. `weekdays` and
  `weeklyTarget` are nullable and only meaningful for their cadence type.
- The `uniq_habit_day` unique index is the real guarantee of "one check-in per day" —
  toggling is an insert-or-delete against it. Never rely on app code alone for uniqueness.

## 5. Reactive data flow

```
 DB (expo-sqlite, enableChangeListener: true)
   └─ Drizzle db.select()/query
        └─ useLiveQuery(...)  ← re-renders on any write
             └─ src/data hook (useTodayHabits, etc.)
                  └─ Screen in app/
```

The DB is opened **once** with change listeners on:

```ts
// src/db/client.ts
import { openDatabaseSync } from 'expo-sqlite';
import { drizzle } from 'drizzle-orm/expo-sqlite';
import * as schema from './schema';

export const expo = openDatabaseSync('happit.db', { enableChangeListener: true });
export const db = drizzle(expo, { schema });
```

Migrations run once at the root layout, gating the app UI:

```ts
// app/_layout.tsx (sketch)
import { useMigrations } from 'drizzle-orm/expo-sqlite/migrator';
import migrations from '../src/db/migrations/migrations';
import { db } from '../src/db/client';

const { success, error } = useMigrations(db, migrations);
// show splash while !success; show error screen on error; render app when success
```

## 7. Domain rules — the parts that are easy to get wrong

These are the rules the whole app's correctness rests on. They live in `src/domain/` as
pure functions and **must** have unit tests.

### 7.1 "Today" and calendar days

- A day is the user's **local** calendar day. Compute it as
  `format(now, 'yyyy-MM-dd')` using the device timezone. This string is the `day` key.
- All comparisons happen on these `YYYY-MM-DD` strings, which sort lexicographically ==
  chronologically. We never compare `Date` objects for "same day" (avoids DST/UTC bugs).

### 7.2 Is a habit scheduled on a given day? — `isScheduledOn(cadence, day)`

- `daily` → always true.
- `weekdays` → true iff `day`'s weekday ∈ `weekdays`. Weekday numbering: **0 = Sunday …
  6 = Saturday** (matches `date-fns` `getDay`). The user's *week-start preference* affects
  only display, never this computation.
- `weekly_count` → **every day is a valid opportunity** (true), but the *goal* is N per
  the ISO week. Scheduling ≠ goal; see streak rules.

### 7.3 Streaks — `computeStreak(checkins, cadence, today)`

A streak counts **consecutive scheduled days that were satisfied**, walking backwards from
today. Precise rules:

- **daily / weekdays:** Walk back day by day. Skip days where `isScheduledOn` is false
  (they don't break the streak). For each scheduled day, it's "satisfied" iff a check-in
  exists for that `day`. Stop at the first *scheduled, unsatisfied* day.
  - **Today grace:** if today is scheduled but not yet checked in, the streak is **not
    broken** — it just doesn't include today. Start the walk at yesterday. (So an unmarked
    "today" never shows a scary 0.) A scheduled *yesterday* with no check-in **does** break it.
- **weekly_count:** Operate on **ISO weeks** (Mon–Sun by ISO, independent of display
  week-start). A week is "satisfied" iff `count(checkins in that week) >= weeklyTarget`.
  Walk back week by week from the current week. The **current week is grace**: it doesn't
  break the streak even if not yet met; count it only once met. Streak unit = weeks.
- **Best streak** is the max streak over all history using the same rules — computed in
  `stats.ts`, not stored.

### 7.4 Toggling a check-in — `toggleCheckin(habitId, day)`

- If a row exists for `(habitId, day)` → delete it. Else insert one. The unique index makes
  this safe against double-taps/races.
- Toggling is only ever allowed for **today** by default in the Today screen. The habit-
  detail calendar *may* allow backfilling past days (v1: allow tapping past days to
  add/remove; never future days).

### 7.5 Timezone / travel edge cases (the acceptance bar)

- Data is timezone-agnostic (`YYYY-MM-DD`), so flying across zones just changes what "today"
  resolves to. A check-in made in Tokyo stays on its Tokyo calendar day; it is **not**
  retroactively shifted. This is the intended, documented behavior.
- DST transitions never affect day math because we never do hour arithmetic on days.

## 8. Error & empty states (must exist, not TODOs)

- **Migration failure:** full-screen recoverable error with the message; never a white
  screen.
- **No habits yet:** Today & Habits show an expressive empty state with a primary
  "Add your first habit" action.
- **All done today:** Today shows a celebratory completed state (see [ui-rules.md](./ui-rules.md) §motion).

## 9. Performance budget

- Cold start to interactive Today screen: **< 1.5s** on a mid-range device.
- Streak/stats recompute for a habit's full history must be **O(history)** and run off the
  render path if history is large (memoize per `(habitId, today, checkinsVersion)`).
